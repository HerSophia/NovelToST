# v3.3 设计增量：Workbench 对话体验统一（Foundation / Worldbuilding / Outline）

> 面向项目：NovelToST  
> 关联基线：v3.2 → v3.3  
> 文档状态：Draft

---

## 0. 版本说明

v3.3 的目标是统一 Workbench 三个工坊的对话体验。范围覆盖：

- 故事基底（Foundation）
- 设定工坊（Worldbuilding）
- 大纲工坊（Outline）

本轮是交互层和前端状态层的统一，不改各工坊的核心业务语义。

### 0.1 本轮已确认决策

根据本轮确认，先冻结以下决策：

1. **统一工作一起做**：交互统一与“回合操作能力”一起推进，不拆成两期。
2. **Foundation 的 `@mention` 暂不纳入 v3.3**：放到下一阶段评估与实施。
3. **Foundation 允许保持单会话模型**：但必须补齐“重试上一次 / 删除上一次”操作。

### 0.2 不变约束

v3.3 需要继续保持以下约束：

1. Foundation 解析失败时，不覆盖现有表单。
2. Outline 解析失败时，保留当前结构化数据（已保留当前结构化数据）。
3. Worldbuilding 解析失败时，不覆盖当前草案。
4. `missingDetailPolicy: 'warn_fallback' | 'strict_block'` 行为不变。
5. Outline 现有 mention 注入限制不变：
   - `OUTLINE_MENTION_PROMPT_MAX_COUNT = 5`
   - `OUTLINE_MENTION_PROMPT_MAX_ITEM_LENGTH = 800`
   - `OUTLINE_MENTION_PROMPT_MAX_TOTAL_LENGTH = 3000`
6. 不破坏现有测试依赖的数据选择器：
   - `data-workbench-*`
   - `data-foundation-*`
   - `data-outline-*`
   - `data-worldbuilding-*`

## 1. 现状与问题

### 1.1 当前三套对话流

1. Foundation：单线程消息流，结构较轻。
2. Outline：会话侧栏 + 消息预览 + 重试/删除上次 + 快照应用，能力完整。
3. Worldbuilding：会话/版本协作能力较强，但回合操作与交互细节未完全对齐 Outline。

### 1.2 当前主要不一致点

1. 输入区行为不一致（快捷键、按钮状态、忙碌提示）。
2. 回合操作不一致（重试上次、删除上次并非三处齐平）。
3. 反馈区域不一致（解析告警展示位置、文案、层级不同）。
4. 消息展示模型不一致（结构化反馈、warnings、snapshot 提示颗粒度不同）。
5. 会话区存在差异（Foundation 可保持单会话，但表现层需要统一节奏）。

## 2. 设计目标与非目标

### 2.1 设计目标

1. 统一三工坊对话交互节奏：输入、发送、处理中、反馈、后续操作。
2. 统一三工坊回合操作能力：至少覆盖“重试上一次 / 删除上一次”。
3. 统一前端展示状态契约，减少重复实现。
4. 保留工坊差异能力，不做功能削平。
5. 在不破坏现有测试钩子的前提下完成改造。

### 2.2 非目标

1. 本轮不引入 Foundation `@mention`。
2. 本轮不重构 Foundation 为多会话持久化模型。
3. 本轮不改变 AI 服务的业务意图分类。
4. 本轮不改动已有安全回退策略和提示主语义。

---

## 3. 统一交互框架

### 3.1 统一布局骨架

三工坊对话区统一为五段结构：

1. 会话区（可选；Foundation 可隐藏或固定单会话）
2. 消息区
3. 输入区
4. 回合操作区（重试上一次 / 删除上一次）
5. 反馈区（解析结果、警告、保留说明）

### 3.2 统一回合操作

统一定义两个基础操作：

1. **重试上一次**：复用上一次用户输入与上下文。
2. **删除上一次**：删除最后一组用户+助手消息回合，并回滚对应展示状态。

统一行为约束：

- 正在请求时禁用回合操作。
- 没有可操作回合时，按钮置灰。
- 操作后消息区、反馈区、预览区同步刷新。

### 3.3 统一发送与忙碌反馈

建议统一：

1. 输入区默认支持：`Ctrl/Cmd + Enter` 发送。
2. 发送后按钮进入忙碌态，统一显示当前动作文本。
3. 忙碌态期间输入区可编辑策略保持一致（建议允许编辑但不可重复发送）。

### 3.4 统一反馈区层级

反馈区统一三层信息：

1. **结果层**：本轮是否成功写入结构化数据。
2. **告警层**：解析不完整、字段忽略、部分保留。
3. **安全层**：明确“当前结构化数据不会被覆盖”的保护语义。

## 4. 统一状态契约（前端展示层）

本轮建议新增“展示层契约”，不强制改底层 store 数据结构。

### 4.1 建议的共享状态字段

```ts
type ConversationViewState = {
  messageInput: string;
  aiBusy: boolean;
  aiBusyAction: string | null;
  lastParseWarning: string | null;
  canRetryLastRound: boolean;
  canDeleteLastRound: boolean;
};
```

### 4.2 建议的消息展示模型

```ts
type ConversationMessageVM = {
  id: string;
  role: 'user' | 'assistant' | 'system';
  text: string;
  createdAt: string;
  parseError?: string | null;
  parseWarnings?: string[];
  rawResponse?: string;
};
```

### 4.3 建议的回合展示模型

```ts
type ConversationRoundStatus = 'applied' | 'partial' | 'failed';

type ConversationRoundVM = {
  roundId: string;
  userMessageId: string;
  assistantMessageId: string;
  status: ConversationRoundStatus;
  warningText?: string | null;
};
```

### 4.4 能力开关

为避免不同工坊硬编码分支，建议引入能力开关：

```ts
type ConversationCapabilities = {
  hasSessions: boolean;
  hasMentions: boolean;
  canRetryLastRound: boolean;
  canDeleteLastRound: boolean;
  hasSnapshotPreview: boolean;
};
```

在 v3.3 中建议默认值：

- Foundation：`hasSessions=false`（或 fixed single session），`hasMentions=false`，`canRetry=true`，`canDelete=true`
- Outline：`hasSessions=true`，`hasMentions=true`，`canRetry=true`，`canDelete=true`
- Worldbuilding：`hasSessions=true`，`hasMentions=true`，`canRetry=true`，`canDelete=true`

## 5. 架构方案：共享对话壳 + 工坊适配器

### 5.1 共享组件

建议抽取以下共享组件（命名可调整）：

1. `WorkbenchConversationShell.vue`
2. `WorkbenchConversationMessages.vue`
3. `WorkbenchConversationComposer.vue`
4. `WorkbenchConversationActions.vue`
5. `WorkbenchConversationFeedback.vue`

### 5.2 工坊适配器接口

建议定义统一适配接口，负责把工坊差异下沉：

```ts
interface WorkbenchConversationAdapter {
  getCapabilities(): ConversationCapabilities;
  getViewState(): ConversationViewState;
  getMessages(): ConversationMessageVM[];
  send(message: string): Promise<void>;
  retryLastRound(): Promise<void>;
  deleteLastRound(): void;
}
```

说明：

- Outline 与 Worldbuilding 可直接映射现有能力。
- Foundation 通过“单会话适配器”补齐重试/删除，不强制多会话。

### 5.3 Foundation 单会话适配策略

Foundation 在 v3.3 采用“固定会话壳”策略：

1. 不新增会话侧栏。
2. 内部以固定会话上下文组织回合。
3. 对外暴露统一回合操作能力。

这样可以在不重构数据结构的前提下，对齐交互行为。

## 10. 风险与缓解

### 风险 1：共享壳抽取后，局部交互细节回归

缓解：

1. 先以 Outline 为基线抽壳。
2. 每接入一个工坊即做回归。

### 风险 2：Foundation 单会话补回合操作时，出现边界问题

缓解：

1. 以“最后一对 user+assistant 消息”为回合边界。
2. 删除与重试逻辑以消息 id 顺序严格判定。

### 风险 3：测试断裂

缓解：

1. 旧选择器保留。
2. 先补统一测试，再迁移各工坊测试。

### 风险 4：后续 mention 接入成本

缓解：

1. v3.3 先在适配器层预留 `hasMentions` 能力位。
2. v3.4 再接入 Foundation mention，不动壳组件结构。

## 11. 验收标准

满足以下条件，可视为 v3.3 完成：

1. 三工坊对话区呈现统一的输入、发送、忙碌、反馈节奏。
2. 三工坊均可执行“重试上一次 / 删除上一次”。
3. Foundation 仍可保持单会话，不影响整体统一体验。
4. 三工坊解析失败时，均明确提示且不覆盖当前结构化数据。
5. 现有关键 data-* 选择器不失效。
6. 工作台与帮助相关单测通过。

## 13. 小结

v3.3 的核心是：

- 把三条对话流的交互手感统一起来。
- 同时补齐基础回合操作能力。
- 保留各工坊业务差异。
- 为下一阶段的 Foundation mention 留好接口。

这样可以在改动风险可控的前提下，先完成用户最直观的统一体验。
