# v3.4 设计增量：设定工坊角色设定档案（CharacterProfile）与角色关系
>
> 面向项目：NovelToST  
> 关联基线：v3.3 → v3.4  
> 文档状态：Draft

## 0. 版本说明

v3.4 的目标是在设定工坊中为 character 类型引入结构化的角色设定档案（CharacterProfile），并新增独立的角色关系对象（CharacterRelationship）。
本轮的核心思路：人物设定不是标签卡片，而是一个可驱动情节的行为模型。每个字段都应当能推导出行为，而不只是描述"这个人是什么样的”。
一个可用的人物设定，至少要能回答以下问题：

- 他在这个阶段想要什么
- 他为什么得不到
- 他会怎么争取
- 他在什么情况下会失控、让步、撒谎、攻击、依赖
- 他会如何影响别人，也如何被别人改变

### 0.1 本轮已确认决策

1. 方案选择：在设定工坊（Worldbuilding）内扩展，给 character 类型加专用结构。不新建独立 Store。
2. Foundation 不动：故事基底的 FoundationProtagonist、FoundationAntagonist、FoundationKeyCharacter 保持现有轻量结构，不升级为九维度模型。
3. **typeProfile 统一字段**：在 `WorldbuildingDraftVersion` 上用一个统一字段 `typeProfile?` 存储专用档案，而不是为每种设定类型单独加一个可选字段。当前只实现 `kind: 'character'` 分支，后续扩展时只需继续增加分支，不必改动 `WorldbuildingDraftVersion` 本身。
4. 关系对象独立存储：不塞在角色内部，放在 WorldbuildingState 顶层。关系不参与 DraftVersion 的版本管理。
5. 动态状态跟着大纲走：人物的当前目标、当前压力、当前信任度等动态数据不存在角色对象里，而是由大纲按章节管理。
6. 常用 / 完整两种模式：数据结构始终完整，UI 控制哪些字段在常用模式下展示。
7. Profile 和 Draft 独立编辑：MVP 阶段不做两者之间的强制同步。AI 协作时同时看到 draft 和 profile，同时输出两者的更新。
8. **Profile 是完整角色模型**：Profile 不仅包含行为驱动字段，也包含描述性字段（外貌、说话方式、背景等），通过 `surface` 分组承载。不单独引入第三层描述结构。

### 0.2 不变约束

1. Foundation 数据结构不改。
2. 通用 WorldbuildingDraft（name / aliases / summary / facts / constraints / relations / extra）保留，继续服务于世界书候选词条生成。
3. 现有 Worldbuilding 的版本管理、会话管理、锁字段、候选词条生成等机制不变。
4. 不破坏现有测试依赖的数据选择器。

## 1. 现状与问题

### 1.1 当前人物数据的分布

人物数据分散在三个地方：

1. Foundation：主角有 13 个字段（name / identity / visibleGoal / deepNeed / coreDesire / coreFear / coreFlaw / behaviorStyle / moralLeaning / mostCaredAbout / bottomLine / temptation / arcDirection），方向偏行为驱动，但覆盖面不够。对手只有 3 个字段，关键角色只有 4 个字段。
2. Worldbuilding：WorldbuildingDraft 是完全通用的结构，不区分设定类型。角色信息全部压缩在 summary（自由文本）、facts（字符串列表）和 extra（自由字典）里，没有结构化的人物模型。
3. Outline：只通过字符串引用角色名字（linkedCharacters、characters），不持有人物数据。

### 1.2 当前的主要问题

1. 设定工坊的角色设定没有结构化字段，AI 协作时只能输出自由文本，无法精确覆盖人物的各个维度。
2. Foundation 的主角模型缺少防御机制、关系模式、错误信念、压力反应、知识状态等维度。
3. 角色之间的关系信息只有两个字符串字段（target + relation），无法描述关系的张力、互动模式和发展方向。
4. 人物没有"变化路径"（弧光）的结构化描述，AI 难以判断人物应该成长、坠落还是固化。
5. 差的字段（如"高冷"“聪明”“温柔”）无法直接指导细纲生成；好的字段（如"被逼到角落时会先切断关系再独自承担"）才能让模型知道戏怎么写。

## 2. 设计目标与非目标

### 2.1 设计目标

1. 为设定工坊的 `character` 类型引入结构化的角色设定档案，覆盖 identity / drive / wound / behavior / relationshipModel / arc 六个核心维度，knowledge / plotConstraints 两个扩展维度，以及 surface 一个描述维度。
2. 新增独立的角色关系对象，支持描述两个角色之间的结构性模式。
3. 档案字段分为常用模式和完整模式，降低用户上手门槛。
4. AI 协作时能同时读写 draft 和 profile，产出的内容可以直接填入结构化字段。
5. 档案跟随现有版本管理机制，支持版本切换和回滚。
6. 使用统一的 `typeProfile` 字段，使后续为其他设定类型加专用结构时不必改动 `WorldbuildingDraftVersion`。

### 2.2 非目标

1. 本轮不改 Foundation 的数据结构。
2. 本轮不为 faction / location / system 等类型做专用结构（但架构上预留扩展点）。
3. 本轮不做 Profile 和 Draft 之间的自动同步。
4. 本轮不在角色对象上存储动态状态（当前目标、信任度等），这些跟大纲走。
5. 本轮不改大纲工坊的数据结构。
6. 本轮不为关系对象引入独立的版本历史。

## 3. 架构方案

### 3.1 typeProfile 统一承载专用结构

不再使用 `characterProfile?` 这种“每种类型一个字段”的写法。

改为在 `WorldbuildingDraftVersion` 上放一个统一字段：`typeProfile?`。

```typescript
export type WorldbuildingTypeProfile =
  | { kind: 'character'; data: CharacterProfile };

export type WorldbuildingDraftVersion = {
  id: string;
  version: number;
  draft: WorldbuildingDraft;
  typeProfile?: WorldbuildingTypeProfile;
  lockedFields: string[];
  createdAt: string;
};
```

当前只实现 `kind: 'character'`。以后如果 `faction`、`location`、`system` 也要加专用结构，只需要在 `WorldbuildingTypeProfile` 里增加分支，不必再改 `WorldbuildingDraftVersion` 本身。

### 3.2 这样做的原因

1. 一个版本只会对应一种专用结构，这个约束在类型上更清楚。
2. 以后扩展其他设定类型时，不必每次都改 `WorldbuildingDraftVersion`。
3. `schema`、`normalize`、`clone` 都更容易收口。
4. 现有通用 Draft 流程不需要推倒重来。

### 3.3 Profile 和 Draft 的分工

- `profile`：结构化角色模型，给编辑、AI 推理、后续剧情驱动使用。
- `draft`：通用文本层，给世界书候选词条、导出和通用流程使用。

这里需要明确一件事：

`profile` 不再只是纯行为层。它会同时包含两类信息：

1. 行为驱动信息，例如目标、错误信念、防御机制、关系模式、弧光。
2. 描述性信息，例如外貌、说话方式、背景、习惯。

也就是说，Profile 在本轮里的定位是**完整角色模型**，而不是只放“会怎么做”的字段。

> MVP 阶段，`profile` 和 `draft` 独立编辑，不做强制同步。
> 后续版本可以考虑提供辅助能力，例如“根据 profile 生成 draft summary”。

### 3.4 关系对象的存放位置

关系对象继续放在 `WorldbuildingState` 顶层：

```text
WorldbuildingState
  ├── sessions: WorldbuildingSession[]
  ├── relationships: CharacterRelationship[]
  ├── candidates: ...
  └── ...
```

关系不属于某一个角色会话，而是跨会话的独立对象。

这里也一并定下一个规则：

- `relationships` **不参与** `DraftVersion` 的版本切换和回滚。

这样做的原因很简单：关系是跨会话数据，硬塞进某个角色版本，会带来归属和回滚范围的问题。本轮先不做关系版本史。

### 3.5 锁字段路径规则

`lockedFields` 使用逻辑路径，不直接暴露内部的 `typeProfile.data` 结构。

推荐写法：

```text
draft.name
draft.summary
profile.identity.name
profile.drive.misbelief
profile.behavior.defenseMechanisms
profile.surface.appearance
```

不推荐写法：

```text
typeProfile.data.drive.misbelief
```

这样做的好处是：UI、AI 和 Store 外层逻辑都不用直接依赖内部包裹结构。

### 3.6 角色名的一致性规则

角色名会同时出现在几个地方：

- `session.title`
- `draft.name`
- `profile.identity.name`

本轮不做强制同步，但需要定义一条软规则：

- 对 `character` 会话，`draft.name` 和 `profile.identity.name` 原则上应保持一致。
- 如果后面增加一致性检查，名字不一致应当列为提示项。

## 4. 角色设定档案类型定义

### 4.1 总结构

角色设定档案分为 6 个核心分组、2 个扩展分组和 1 个描述分组：

| 分组 | 职责 | 类别 |
| ------ | ------ | ------ |
| identity | 这个人是谁 | 核心 |
| drive | 他想要什么，缺什么，怕什么 | 核心 |
| wound | 他为什么会变成这样 | 核心 |
| behavior | 他遇事怎么做 | 核心 |
| relationshipModel | 他如何进入和处理关系 | 核心 |
| arc | 他会怎么变，不能怎么变 | 核心 |
| knowledge | 他知道什么，不知道什么 | 扩展 |
| plotConstraints | 剧情约束，给系统使用 | 扩展 |
| surface | 他看起来是什么样，怎么说话 | 描述 |

### 4.2 完整类型定义

```typescript
// ===== 1. identity：这个人是谁 =====
export type CharacterIdentity = {
  name: string;
  role: string;                    // 故事职能：主角、对手、导师、镜像角色、推动者等
  publicIdentity: string;          // 公开身份：独立记者、退役军官
  hiddenIdentity: string;          // 隐藏身份：旧案幸存者家属
  camp: string;                    // 阵营 / 立场
  socialPosition: string;          // 社会位置
  storyFunctions: string[];        // 叙事功能标签：调查者、关系线核心
};

// ===== 2. drive：想要什么，缺什么，怕什么 =====
export type CharacterDrive = {
  externalGoal: string;            // 外在目标（长期）
  coreNeed: string;                // 内在需求（人物弧光核心）
  misbelief: string;               // 错误信念
  coreFear: string;                // 核心恐惧
  midTermGoal: string;             // 中期目标
  shortTermGoalPatterns: string[]; // 常见的短期目标模式
  bottomLines: string[];           // 底线
  canCompromiseOn: string[];       // 可以妥协的事
  cannotCompromiseOn: string[];    // 不可妥协的事
};

// ===== 3. wound：为什么会变成这样 =====
export type CharacterWound = {
  deepWound: string;               // 深层创伤
  shameTriggers: string[];         // 最容易被刺痛的点
  taboos: string[];                // 禁忌 / 绝不能触碰的事
};

// ===== 4. behavior：遇事怎么做 =====
export type CharacterBehavior = {
  baselineState: string;           // 基线状态：平时是什么样
  decisionStyle: string;           // 决策风格
  stressResponse: string;          // 压力反应
  conflictStyle: string;           // 冲突处理方式
  defenseMechanisms: string[];     // 惯用防御机制
  carePattern: string;             // 关心他人的方式
  deceptionStyle: string;          // 隐瞒 / 欺骗方式
  persuasionStyle: string;         // 说服他人的方式
  underPressureActions: string[];  // 高压下的典型行为
  failurePattern: string;          // 失败时的行为模式
};

// ===== 5. relationshipModel：如何进入和处理关系 =====
export type CharacterRelationshipModel = {
  attachmentStyle: string;         // 依恋类型
  trustPattern: string;            // 信任建立方式
  intimacyPattern: string;         // 亲密关系模式
  powerPattern: string;            // 权力 / 支配模式
  jealousyPattern: string;         // 嫉妒表现方式
};

// ===== 6. arc：会怎么变，不能怎么变 =====
export type CharacterArc = {
  startState: string;              // 初始状态
  midShift: string;                // 中段动摇点
  endState: string;                // 最终状态
  unchangedCore: string;           // 不变核心
  negativeArcPossible: string;     // 可能的崩坏 / 坠落路径
  mustNotBreak: string[];          // 绝不能破的约束
};

// ===== 7. knowledge：知道什么，不知道什么 =====
export type CharacterKnowledge = {
  knownFacts: string[];            // 故事开始时已知的事实
  blindSpots: string[];            // 盲区 / 误判
  secretsKept: string[];           // 保守的秘密
};

// ===== 8. plotConstraints：剧情约束 =====
export type CharacterTurningCondition = {
  change: string;                  // 变化描述
  requires: string[];              // 前置条件
};

export type CharacterPlotConstraints = {
  mustDo: string[];                // 必须做的事
  mustNotDo: string[];             // 绝不能做的事
  turningConditions: CharacterTurningCondition[];
};

// ===== 9. surface：外在表现 =====
export type CharacterSurface = {
  age: string;                     // 年龄或年龄段
  appearance: string;              // 外貌 / 体态
  speechStyle: string;             // 说话方式
  background: string;              // 背景简史
  habits: string[];                // 日常习惯 / 偏好
  signatureTraits: string[];       // 标志性特征
};

// ===== 角色设定档案（完整） =====
export type CharacterProfile = {
  identity: CharacterIdentity;
  drive: CharacterDrive;
  wound: CharacterWound;
  behavior: CharacterBehavior;
  relationshipModel: CharacterRelationshipModel;
  arc: CharacterArc;
  knowledge: CharacterKnowledge;
  plotConstraints: CharacterPlotConstraints;
  surface: CharacterSurface;
};
```

### 4.3 字段设计原则

行为相关的字段，应当尽量能推出人物会怎么做。

不好的写法：

- 高冷
- 聪明
- 温柔
- 疯批

这类词很难直接指导情节。

较好的写法：

- 被逼到角落时，会先切断关系再独自承担
- 被看穿脆弱时会攻击对方
- 不会直接求助，只会以交易包装请求
- 在关系里通过行动示好，但拒绝口头确认

这类内容能直接帮助 AI 判断人物在场景中的反应。

`surface` 组不负责推动行为。它的作用是补足人物的外在信息，方便世界书和场景描写使用。

### 4.4 为什么不单独设置“喜好”“性格标签”等字段

本设计不单独增加“喜好”“性格”“口头禅”这类通用字段，原因如下：

1. 这类字段大多不能直接推动情节。
2. 如果某个偏好会影响行为，它通常已经能放进 `drive`、`wound`、`behavior` 这些分组里。
3. 如果只是帮助描写，例如外貌、说话方式、日常习惯、标志性动作，那么放进 `surface` 即可。
4. 一旦单独设置“性格”字段，用户很容易填成“高冷”“温柔”这类标签，会偏离本轮的目标。

简单说：

- 能影响行为的，放行为字段。
- 只帮助描写的，放 `surface`。
- 无法进入这两类的，再考虑放 Draft 的 `facts`。

## 5. 常用模式与完整模式

数据结构始终完整，UI 只控制展示范围。

### 5.1 常用模式（约 14 个字段）

| 分组 | 字段 |
| ------ | ------ |
| identity | name, role, publicIdentity |
| drive | externalGoal, coreNeed, misbelief, coreFear, bottomLines[] |
| wound | deepWound |
| behavior | decisionStyle, stressResponse, defenseMechanisms[] |
| arc | startState, endState |

说明：`defenseMechanisms` 放在常用模式，是因为它对冲突写作的帮助很直接。

### 5.2 完整模式

在常用模式基础上，再展开以下字段：

| 分组 | 新增字段 |
| ------ | ---------- |
| identity | hiddenIdentity, camp, socialPosition, storyFunctions[] |
| drive | midTermGoal, shortTermGoalPatterns[], canCompromiseOn[], cannotCompromiseOn[] |
| wound | shameTriggers[], taboos[] |
| behavior | baselineState, conflictStyle, carePattern, deceptionStyle, persuasionStyle, underPressureActions[], failurePattern |
| relationshipModel | attachmentStyle, trustPattern, intimacyPattern, powerPattern, jealousyPattern |
| arc | midShift, unchangedCore, negativeArcPossible, mustNotBreak[] |
| knowledge | knownFacts[], blindSpots[], secretsKept[] |
| plotConstraints | mustDo[], mustNotDo[], turningConditions[] |
| surface | age, appearance, speechStyle, background, habits[], signatureTraits[] |

说明：

- `relationshipModel` 整组放在完整模式。它对关系线很重要，但不是每个角色都必须先填。
- `surface` 整组放在完整模式。它主要帮助描写和世界书整理，不是剧情驱动字段中的最高优先级。

## 6. 角色关系对象

### 6.1 角色引用类型

角色引用不再使用 `'name:角色名'` 这种字符串写法，改为明确的对象结构：

```typescript
export type CharacterRef = {
  refType: 'session' | 'name';
  sessionId?: string;
  name: string;
};
```

含义如下：

- `refType: 'session'`：说明这个角色已经有设定工坊会话，`sessionId` 是主标识，`name` 用于显示。
- `refType: 'name'`：说明这个角色还没有设定工坊会话，只能先按名字引用。

### 6.2 关系类型定义

```typescript
export type CharacterRelationship = {
  id: string;
  characterA: CharacterRef;
  characterB: CharacterRef;
  publicState: string;             // 公开关系状态
  coreStateAtoB: string;           // A 对 B 的核心态度
  coreStateBtoA: string;           // B 对 A 的核心态度
  coreTension: string[];           // 核心张力
  interactionPattern: string[];    // 互动模式
  milestones: string[];            // 关系里程碑
  updatedAt: string;
};
```

### 6.3 角色标识策略

角色标识采用“会话优先，名字兜底”的方式：

- 如果角色已经有设定工坊会话，使用 `refType: 'session'`。
- 如果角色还没有会话，使用 `refType: 'name'`。
- 当一个只有名字的角色后来建立了会话，系统应提供迁移操作，把它改成 `refType: 'session'`。

### 6.4 关系对象不包含动态数值

关系对象只描述两个角色之间相对稳定的结构信息，例如：

- 核心态度
- 核心张力
- 互动模式
- 关系里程碑

以下内容不放在关系对象中，而是跟着大纲按章节管理：

- 当前信任度
- 当前欲望度
- 当前情绪负荷
- 当前关系阶段

这样做的原因是：这些值会随着情节推进不断变化，放在大纲里更稳妥。

### 6.5 `draft.relations` 和顶层 `relationships` 的分工

现在会同时存在两种关系数据：

- `relationships`：结构化关系主模型，用来存放完整的关系信息。
- `draft.relations`：通用的轻量关系索引，继续服务于世界书候选词条和通用流程。

对角色会话来说，`draft.relations` 不是正式的关系模型，只是简化引用。

MVP 阶段，两者不做自动双向同步。

### 6.6 关系不参加版本回滚

`typeProfile` 跟随 `DraftVersion` 走，所以角色档案可以随版本切换和回滚。

`relationships` 放在 `WorldbuildingState` 顶层，不跟着 `DraftVersion` 回滚。

这是本轮的明确取舍。原因有两点：

1. 关系是跨会话对象，不适合挂在某一个角色版本下。
2. 如果本轮就加入关系版本史，复杂度会明显上升。

### 6.7 删除角色会话时的处理

如果某个关系对象引用了一个 `sessionId`，而这个会话后来被删除：

- 不自动删除关系对象。
- 将对应的 `CharacterRef` 从 `refType: 'session'` 降级为 `refType: 'name'`。
- `name` 使用删除前最后一次可读到的角色名。

这样可以尽量保留已有的关系信息。

## 7. 稳定层与动态层

### 7.1 稳定层（存在角色对象中）

很少变化，属于人物底层结构：

- 核心需求、错误信念、深层伤口、核心恐惧
- 防御机制、决策风格、冲突方式
- 依恋类型、信任模式、亲密模式
- 底线、禁忌
- 弧光路径

### 7.2 动态层（跟着大纲走）

随剧情推进会更新：

- 当前阶段目标
- 当前压力源
- 当前对某人的信任度
- 当前情绪负荷
- 已知信息
- 当前立场
- 当前关系状态

本轮不定义动态层的类型结构。动态层的设计将在大纲工坊的后续版本中一并处理。

## 8. 对现有类型的改动

### 8.1 WorldbuildingDraftVersion

`WorldbuildingDraftVersion` 不再直接挂 `characterProfile?`，改为统一的 `typeProfile?`：

```typescript
export type WorldbuildingTypeProfile =
  | { kind: 'character'; data: CharacterProfile };

export type WorldbuildingDraftVersion = {
  id: string;
  version: number;
  draft: WorldbuildingDraft;
  typeProfile?: WorldbuildingTypeProfile;
  lockedFields: string[];
  createdAt: string;
};
```

说明：

- 当前只有 `kind: 'character'`。
- 以后如果其他设定类型也要加专用结构，只需要继续扩展 `WorldbuildingTypeProfile`。
- `WorldbuildingDraftVersion` 本身不必反复改动。

### 8.2 WorldbuildingState

`WorldbuildingState` 新增 `relationships` 字段：

```typescript
export type WorldbuildingState = {
  sessions: WorldbuildingSession[];
  activeSessionId: string | null;
  relationships: CharacterRelationship[];
  candidates: WorldbookEntryCandidate[];
  selectedWorldbookName: string | null;
  commitMode: WorldbookCommitMode;
  updatedAt: string;
};
```

### 8.3 Foundation 不改

`FoundationProtagonist`、`FoundationAntagonist`、`FoundationKeyCharacter` 保持现状。

它们仍然是故事基底层的轻量索引。完整的人物模型放在设定工坊的 `profile` 中。

## 9. AI 协作变更

### 9.1 AI 看到什么

当会话类型为 `character` 时，AI 提示词需要同时包含：

1. 当前 draft
2. 当前 profile

这里有一个约定：

- **内部存储** 使用 `typeProfile`
- **AI 输入输出** 统一使用 `profile`

这样可以避免让 AI 直接依赖内部包装结构。

### 9.2 AI 输出格式

角色会话的 AI 输出格式调整为：

```json
{
  "assistantReply": "本轮我补全了角色的错误信念和防御机制。",
  "draft": {
    "name": "",
    "aliases": [],
    "summary": "",
    "facts": [],
    "constraints": [],
    "relations": [],
    "extra": {}
  },
  "profile": {
    "identity": { "...": "..." },
    "drive": { "...": "..." },
    "wound": { "...": "..." },
    "behavior": { "...": "..." },
    "relationshipModel": { "...": "..." },
    "arc": { "...": "..." },
    "knowledge": { "...": "..." },
    "plotConstraints": { "...": "..." },
    "surface": { "...": "..." }
  },
  "consistencyIssues": []
}
```

解析时，再根据会话类型把 `profile` 包装成内部的 `typeProfile`：

```typescript
typeProfile = {
  kind: 'character',
  data: profile,
};
```

### 9.3 解析逻辑

`extractDraftPayload` 之外，需要新增 `extractProfilePayload`，负责从 AI 返回的 JSON 中提取 `profile`。

解析失败时的规则不变：

- 不覆盖现有 profile
- 保留当前版本数据

### 9.4 AI 提示词方向

针对角色类型，提示词需要强调：

1. 行为字段尽量写成可推导行为的内容，不写标签式描述。
2. 默认优先补常用模式字段，不必一次写满全部字段。
3. `misbelief` 和 `defenseMechanisms` 是关键字段，需要尽量具体。
4. `surface` 字段应服务于外在描写，不要和行为字段混写。
5. 一致性检查重点看 `drive / wound / behavior / arc` 是否互相支撑。

## 10. 世界书候选词条生成

当生成世界书候选词条时，角色类型的会话同时参考 draft 和 profile。

profile 可以贡献的内容包括：

- `identity` 中的身份信息
- `drive` 中的目标和底线
- `behavior` 中的行为模式
- `arc` 中的变化方向
- `knowledge` 中的已知信息和秘密
- `surface` 中的外貌、说话方式、习惯和标志性特征

具体拼接规则在实现阶段再细化，本轮不强行规定。

## 12. Store 层变更要点

### 12.1 校验结构扩展

需要为 `CharacterProfile`、`CharacterRef`、`CharacterRelationship`、`WorldbuildingTypeProfile` 增加校验结构。

默认值规则：

- 字符串字段默认 `''`
- 数组字段默认 `[]`
- 对象字段按空对象展开默认值

### 12.2 DraftVersion 的整理逻辑

`WorldbuildingDraftVersionSchema` 需要新增 `typeProfile` 可选字段。

`normalizeDraftVersion` 中：

- `typeProfile` 存在时，按 `kind` 处理其 `data`
- `typeProfile` 不存在时，保持 `undefined`

`cloneDraftVersion` 中：

- 深拷贝 `typeProfile`

### 12.3 角色会话的默认初始化

新建 `character` 会话时，建议直接初始化空 profile，而不是等到第一次编辑时再创建。

这样做的好处是：

1. UI 不需要处理“角色会话没有 profile”的中间状态。
2. AI 协作时可以直接读取完整结构。
3. 代码分支更少。

### 12.4 Store 方法

Profile 相关：

- `patchTypeProfile(sessionId, profilePatch, versionId?)`
- `replaceTypeProfile(sessionId, profile, versionId?)`

关系相关：

- `addRelationship(input)`
- `patchRelationship(relationshipId, patch)`
- `removeRelationship(relationshipId)`

### 12.5 Patch 规则

建议统一采用以下规则：

- 对象：深层合并
- 数组：整体替换
- 字符串：直接替换

例如：

```typescript
patchTypeProfile(sessionId, {
  drive: {
    misbelief: '依赖别人最终只会失去主动权',
  },
});
```

这只会修改 `drive.misbelief`，不会覆盖 `drive` 里的其他字段。

### 12.6 关系数据的整理与恢复

`normalizeWorldbuildingState` 需要处理 `relationships`。

`hydrate` 和 `toStateSnapshot` 需要包含 `relationships`。

删除角色会话时，需要扫描所有关系引用；如果发现有 `refType: 'session'` 指向被删会话，就按第 6.7 节的规则降级成 `refType: 'name'`。

## 17. 风险与处理

### 风险 1：字段较多，填写压力偏大

处理办法：

1. 默认只展示常用模式。
2. AI 默认优先补常用模式字段。
3. 空字段不影响其他功能。

### 风险 2：AI 仍然把字段写成标签

处理办法：

1. 提示词明确要求写具体反应和具体模式。
2. 给出正反示例。
3. 一致性检查重点看 `drive / wound / behavior / arc` 是否互相支撑。

### 风险 3：关系引用失效

处理办法：

1. 已有会话时优先使用 `sessionId`。
2. 没有会话时保留 `name`。
3. 会话被删除时自动降级引用，不直接删关系。
4. 后续如果需要，再升级成独立角色 ID。

### 风险 4：profile 和 draft 不一致

处理办法：

1. AI 协作时同时看到两者。
2. MVP 阶段不做强制同步，避免把逻辑做得过重。
3. 后续可以增加一致性检查，只做提醒，不强制改写。

### 风险 5：关系不回滚，用户可能误以为是缺陷

处理办法：

1. 在文档中明确写清这个规则。
2. 在 UI 中给出必要提示。
3. 后续如果确实需要，再考虑关系版本史。

## 18. 后续版本方向

1. 在大纲侧管理角色的动态状态，例如当前目标、压力、信任、情绪负荷。
2. 为 `faction`、`location`、`system` 等类型增加自己的 profile 分支。
3. 增加 profile 和 draft 的一致性检查。
4. 打通 Foundation、Worldbuilding、Outline 三层的角色引用。
5. 如果角色数据的重要性继续上升，再考虑独立角色 ID。

### 18.1 这个范式对其他设定类型的参考价值

本轮形成的基本做法，不只适用于角色，也可以作为其他设定类型的参考：

1. 保留通用 `draft`，继续服务于世界书候选词条、导出和通用流程。
2. 用 `typeProfile` 给不同类型挂各自的结构化档案。
3. 把相对稳定的信息放进 profile。
4. 把会随剧情推进不断变化的信息放到 Outline 一侧管理。
5. 把关系对象独立出来，不强行塞进某一个实体内部。

这个做法对 `faction`、`location`、`system` 都有参考价值。

### 18.2 不同类型，稳定层和动态层的边界并不相同

虽然整体做法可以复用，但不同类型的“稳定层”和“动态层”边界并不一样。

#### 18.2.1 faction

`faction` 是最需要注意的一类。

相对稳定的内容通常是：

- 核心理念
- 创立目的
- 组织结构
- 行事风格
- 内部矛盾的根源

变化很快的内容通常是：

- 当前成员构成
- 当前控制范围
- 当前资源状况
- 当前派系力量对比
- 当前对外关系状态

这说明 `faction` 的 profile 更适合描述“这个组织本来是什么样”，而不是“它现在处于什么局面”。

如果后续真的做 `faction`，Outline 一侧需要较早补上对应的动态状态承载能力。否则 profile 会偏空，很多真正影响剧情的信息没有地方放。

#### 18.2.2 location

`location` 的情况比 `faction` 稳定很多。

相对稳定的内容通常是：

- 地理位置
- 空间结构
- 历史背景
- 功能定位
- 氛围和感官特征

动态内容通常是：

- 当前控制者
- 当前开放或封锁状态
- 当前驻扎人物
- 当前损毁情况

这一类比较容易套用本轮的范式。

#### 18.2.3 system

`system` 通常以稳定信息为主，例如：

- 规则
- 限制
- 代价
- 等级结构
- 适用范围

它的动态变化通常不是持续状态，而是剧情中的规则突破、异常事件、新发现。

因此，这一类也比较适合沿用本轮范式。

### 18.3 关系模型以后可能需要继续泛化

本轮只做角色和角色之间的关系，所以当前使用 `CharacterRef` 和 `CharacterRelationship`。

但后续如果扩展到更多设定类型，关系很可能不只发生在角色之间，还会出现：

- 角色和势力的关系
- 势力和势力的关系
- 势力和地点的关系
- 角色和地点的关系

到那时，关系模型可能需要从“角色关系”进一步扩展为“实体关系”。

也就是说：

- 本轮的角色关系模型足够支撑 v3.4
- 但它不应被视为所有类型关系的最终形态

### 18.4 一个明确结论

因此，可以先定下一条判断：

- 当前范式可以作为其他设定类型的基础参考。
- 但对于 `faction` 这类动态变迁很重的类型，真正的难点不在 profile 本身，而在 Outline 一侧是否能承接它的动态状态。
- 如果 Outline 一侧没有相应能力，那么这类类型即使加了 profile，也只能先覆盖“稳定设定”，还不能完整覆盖“当前局势”。

## 20. 验收标准

满足以下条件，可视为 v3.4 完成：

1. `character` 类型会话的 `DraftVersion` 可以存储 `typeProfile`。
2. 角色 profile 跟随版本管理，支持切换和回滚。
3. 锁字段同时覆盖 `draft.*` 和 `profile.*` 路径。
4. AI 协作时，角色类型提示词包含 profile，AI 输出能正确解析并映射为 `typeProfile`。
5. 解析失败时，不覆盖当前 profile 数据。
6. 关系对象支持增删改，独立于会话存储。
7. 关系对象不跟随 `DraftVersion` 回滚，这一规则在文档和界面中有明确说明。
8. UI 支持常用 / 完整模式切换。
9. `surface` 分组能正常编辑和保存。
10. 非角色类型会话不受影响。
11. 旧数据加载时能正常工作，新增字段使用默认值。
12. 现有测试不失效。

## 21. 小结

v3.4 的核心有六点：

- 角色档案继续留在 Worldbuilding 中，不新开存储层。
- 角色档案使用完整 profile，而不是只存几个标签。
- 内部用 `typeProfile` 统一承载专用结构，便于以后继续扩展。
- `draft` 负责通用文本层，`profile` 负责结构化人物模型。
- 关系独立存放在顶层 `relationships`，不塞进角色 profile，也不跟版本回滚。
- 角色的动态状态继续放在 Outline，不放进本轮的角色档案。

这样处理以后，角色设定会更接近“能推动剧情的人”，而不是一组零散标签。
