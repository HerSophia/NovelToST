# v3.1 设计增量：故事基底的分阶层表单

> 面向项目：NovelToST  
> 关联基线：v3（Foundation-first）→ v3.1（Foundation-first with Tiered Editing）  
> 文档状态：Draft

---

## 0. 版本说明

v3 已经把「故事基底」确立为写作流程的起点，也已经落地了 8 个固定模块、对话协作、表单编辑、模块进度追踪。

v3.1 不改这个方向。v3.1 处理的是另一个问题：**当前故事基底虽然完整，但第一次使用时填写压力仍然偏大，尤其不适合只想尽快开始写故事的用户。**

因此，v3.1 的核心目标是：

1. 将故事基底改成**基础 / 中级 / 高级**三层需求。
2. 让一部分表单成为**明确的可选项**。
3. 让分层不仅影响界面，也影响生成前的校验与提醒。
4. 尽量不改 `StoryFoundation` 数据结构，避免波及下游。

这是一份**增量设计**。它建立在 v3 之上，不替换 v3 的基础模型。

### 0.1 决策冻结

本轮先冻结以下决策：

1. **保留 `StoryFoundation` 现有数据结构。** 不新增新的层级对象，不拆分现有模块结构。
2. **新增三层编辑需求。** 分为基础层、中级层、高级层。
3. **默认不展示高级层。** 只有切换到“高级模式”后，才显示高级层模块。
4. **基础层完成后允许直接生成。** 不要求所有模块都填完。
5. **生成前增加软提醒。** 当用户只完成基础层，或基础层已完成但中级层明显不足时，先弹出提醒对话框，再允许继续生成。
6. **分层影响生成校验。** 不再只用“整份基底是否完整”作为判断条件，而是改为“是否达到最低可运行门槛”。
7. **Prompt 注入原则不变。** 仍然只注入已有内容，不为空字段补造默认值。
8. **现有对话协作链路尽量不动。** `foundation-ai.service.ts`、`useFoundationControl.ts`、`StoryFoundation` 存储与持久化链路保持兼容。

## 1. 背景与问题

### 1.1 当前问题

当前故事基底的 8 个模块表达能力完整，但在实际使用中会遇到几个问题：

1. 用户第一次进入页面时，需要面对较多字段。
2. 用户容易分不清哪些字段是“必须先想清楚的”，哪些只是“以后可以再补的”。
3. 如果把所有模块都当作同一层级，轻度用户会觉得成本过高。
4. 如果为了降低门槛而完全放弃校验，生成结果又容易过于发散。

### 1.2 用户侧考虑

本项目并不只面向重度规划型用户，也面向不少只想快速开始、满足个人写作需求的用户。

这意味着产品不能把“完整设定”当作默认前提，而应当允许用户：

1. 先填最少的一组信息。
2. 先开始生成。
3. 在后续使用中再继续补充中级和高级约束。

因此，v3.1 不追求“所有人都一次性写完整”，而是追求“先可用，再逐步变强”。

## 2. 设计目标与非目标

### 2.1 设计目标

1. 降低故事基底的首次填写门槛。
2. 让用户清楚知道：哪些信息会直接影响能否开始生成。
3. 让中级和高级信息成为清晰的增强项，而不是隐性负担。
4. 保持当前 `StoryFoundation` 与下游使用方式基本稳定。
5. 让 FoundationPanel 的改造尽量基于现有模块式结构，而不是整体推倒重来。

### 2.2 非目标

本轮不做以下事情：

1. 不重构 `StoryFoundation` 的对象结构。
2. 不把 8 个模块改成全新的领域模型。
3. 不重写 foundation prompt 的业务语义。
4. 不把分层做成复杂的打分系统。
5. 不引入大量新的流程状态或持久化格式。

## 3. 整体方案

v3.1 采用**“模式切换 + 模块分层 + 生成前提醒”**的组合方案。

### 3.1 用户模式

界面上区分两种模式：

1. **普通模式**
   - 默认模式。
   - 显示基础层与中级层。
   - 不显示高级层的详细表单。

2. **高级模式**
   - 由用户主动切换。
   - 显示全部层级。
   - 允许编辑高级层模块与扩展模块。

### 3.2 模块层级

建议按模块分三层：

| 层级 | 模块 | 说明 |
| --- | --- | --- |
| 基础 | `positioning` | 确定故事是什么 |
| 基础 | `core` | 确定故事讲什么 |
| 基础 | `protagonist` | 确定谁在行动 |
| 中级 | `keyRelations` | 明确关键角色关系 |
| 中级 | `conflictFramework` | 明确冲突如何推进 |
| 中级 | `worldBrief` | 明确故事需要什么样的世界 |
| 高级 | `narrativeRules` | 控制写法、视角、披露方式 |
| 高级 | `endgame` | 控制终局方向 |
| 高级 | `extensions` | 自定义扩展 |

### 3.3 为什么这样分

这个分法不是按字段数量划分，而是按“对起步生成的影响强度”划分。

- 基础层回答最基本的三个问题：这是什么故事、讲什么、谁在推进。
- 中级层主要提升稳定性和一致性。
- 高级层主要提供精细控制，不应阻塞开始使用。

## 5. 生成前校验规则

### 5.1 校验分两层

v3.1 将校验分为两层：

1. **硬性门槛**：不满足则不允许进入生成。
2. **软性提醒**：满足最低门槛后允许进入生成，但先提醒用户还可以继续补充。

### 5.2 最低可运行门槛

建议把最低门槛收敛为以下条件：

1. `positioning.genre` 已填写。
2. `core.logline` 已填写。
3. `protagonist.name` / `protagonist.identity` / `protagonist.visibleGoal` 三者至少填写一项。

满足以上条件，即可视为“基础层已达到最低可运行门槛”。

### 5.3 基础建议项

以下字段不阻塞生成，但会参与提醒：

- `positioning.title`
- `positioning.mainType`
- `core.coreConflict`
- `core.emotionalTone`
- `protagonist.deepNeed`
- `protagonist.coreFlaw`

### 5.4 中级建议项

以下内容也不阻塞生成，但如果整体缺失，系统应在生成前提示用户：

- `keyRelations.antagonist.*`
- `conflictFramework.mainConflict`
- `conflictFramework.failureCost`
- `worldBrief.worldType`
- `worldBrief.requiredRules`

### 5.5 高级层处理方式

高级层完全不作为生成门槛。

也就是说：

1. 未填写高级层时，不阻塞生成。
2. 已填写高级层时，生成时仍然生效。
3. 普通模式下若高级层已有内容，应给出一个简短状态提示，但不强制展开。

## 6. 生成前提醒对话框

### 6.1 触发条件

当用户点击“生成总纲”或其他依赖故事基底的生成入口时：

1. 若未达到最低门槛：直接阻止，并提示缺失项。
2. 若已达到最低门槛，但基础建议项或中级建议项明显不足：弹出提醒对话框。
3. 若用户确认继续：允许生成。

### 6.2 建议文案

#### 标题

已具备生成条件

#### 正文

当前故事基底已经可以支持生成。  
如果继续补充人物关系、冲突结构或世界规则，结果通常会更稳定，也更贴近你的预期。

#### 可附带的缺失提示

建议补充：

- 核心冲突
- 关键对手
- 失败代价
- 世界规则

#### 按钮

- 继续生成总纲 / 继续派生细纲 / 继续重写细纲
- 先去补充

### 6.3 设计原则

这个对话框不是拦截式惩罚，而是温和提示。

目标不是劝退用户，而是让用户知道：

1. 现在已经可以开始。
2. 如果再多填一点，结果会更稳。

### 6.4 当前落实状态

截至当前实现，第三阶段已按以下方式落地：

1. `generationReadiness` 已接入总纲生成、细纲派生、章节细纲重写三个入口。
2. 未达到最低门槛时，系统会直接阻止生成，并提示缺失项。
3. 已达到最低门槛但建议项不足时，系统会弹出工作台内提醒弹窗。
4. 提醒交互不再依赖浏览器 `window.confirm(...)`。
5. 用户确认后继续执行，用户取消后返回故事基底完善流程。

## 7. FoundationPanel 的最小改造方案

v3.1 建议尽量复用现有 `FoundationPanel.vue`、`FoundationModulePanel.vue`、`FoundationProgressBar.vue` 结构。

### 7.1 新增模式切换

在模块编辑区头部新增一个模式切换控件：

- 普通模式
- 高级模式

行为如下：

1. 默认进入普通模式。
2. 普通模式仅显示基础层和中级层模块。
3. 高级模式显示全部模块。
4. 切换模式不影响实际数据，只影响可见性。

### 7.2 普通模式下的模块显示

普通模式下建议显示：

- `positioning`
- `core`
- `protagonist`
- `keyRelations`
- `conflictFramework`
- `worldBrief`

普通模式下隐藏：

- `narrativeRules`
- `endgame`
- `extensions`

### 7.3 高级层已存在数据时的提示

普通模式下，如果高级层存在有效内容，建议在模块编辑区顶部增加一条提示：

- 已存在高级设定，生成时会一并生效。

这条提示只说明状态，不展开全部内容。

### 7.4 模块卡片标记

在 `FoundationModulePanel` 头部建议增加一个层级标记：

- 基础
- 中级
- 高级

这样用户可以直观看出每个模块在体系中的位置。

### 7.5 字段级提示

第一版不建议把字段做成复杂的多层折叠。

更稳的做法是：

1. 先保留当前模块内部字段布局。
2. 对基础门槛字段标注“最低门槛”。
3. 对建议项标注“建议填写”。
4. 其余字段默认视为可选。

这样改动较小，也更容易落地。

### 7.6 快速跳转与筛选

现有“快速跳转”和“模块筛选”建议继续保留，但数据源改为“当前模式下可见模块”。

也就是说：

1. 普通模式下，快速跳转只列出基础层与中级层。
2. 高级模式下，快速跳转列出全部模块。
3. 模块完成状态仍然保留原有 empty / partial / complete。

### 7.7 角色关系与扩展区处理

当前页面中，关键角色编辑区和扩展编辑区在模块区之外单独呈现。

v3.1 建议如下处理：

1. **关键角色编辑区**继续保留在普通模式中，因为它属于 `keyRelations` 的增强部分。
2. **扩展编辑区**只在高级模式中显示，因为它属于高级层。

这样可以减少普通模式下的负担。

## 8. 进度与状态设计

### 8.1 保留原有模块状态

当前 `moduleStatuses` 使用：

- `empty`
- `partial`
- `complete`

v3.1 第一版建议保留这套状态，不引入新的状态值。

### 8.2 新增层级汇总信息

在保留原状态的基础上，新增三类汇总：

1. **基础层完成情况**
   - 例如：3 / 3 已完成
2. **中级层补充情况**
   - 例如：2 / 3 已完善
3. **高级层启用情况**
   - 例如：1 / 3 已填写

### 8.3 进度条展示建议

`FoundationProgressBar` 建议从“8 个模块统一进度”调整为“按层级提示”为主，但不必完全放弃总数统计。

建议第一版文案如下：

- 基础层：已具备生成条件 / 尚未具备生成条件
- 中级层：建议继续补充
- 高级层：按需填写

这样比单纯显示“完成了几个模块”更贴近用户决策。

## 9. 技术设计建议

### 9.1 数据结构保持不变

保留：

- `StoryFoundation`
- `FoundationModuleId`
- 现有 patch 结构
- 现有持久化结构

不建议把层级直接写进 foundation 数据对象本身。

### 9.2 新增元数据配置层

建议新增一份独立元数据配置，作为 UI、校验、进度汇总的共同来源。

建议内容包括：

1. 模块所属层级
2. 字段所属分类（门槛 / 建议 / 可选）
3. 普通模式是否显示
4. 高级模式是否显示
5. 生成最低门槛规则
6. 生成提醒规则

### 9.3 建议新增的共享能力

可以考虑新增一个共享文件，统一提供以下能力：

- `FOUNDATION_MODULE_TIERS`
- `FOUNDATION_FIELD_META`
- `getFoundationGenerationReadiness(foundation)`
- `getFoundationTierSummary(foundation)`
- `hasAdvancedFoundationContent(foundation)`

这样可以避免：

1. UI 自己判断一套
2. 生成入口自己判断一套
3. store 又维护一套单独规则

### 9.4 与现有 `REQUIRED_FIELDS` 的关系

当前 `types/foundation.ts` 中的 `REQUIRED_FIELDS` 更偏向“模块完成判断”。

v3.1 后，建议不要再让它承担全部语义。

更合适的做法是：

1. `REQUIRED_FIELDS` 保留或逐步收缩，只用于旧的模块完成逻辑。
2. 新增更细的 readiness 规则，专门负责生成门槛与提醒。

这样迁移更稳，不会一次性影响太多地方。

## 10. 对下游链路的影响

### 10.1 对 prompt 的影响

原则不变：**只注入已填写内容。**

分层不会改变 prompt 的业务含义，只改变“用户是否必须先填这些字段”。

### 10.2 对世界设定工坊与大纲工坊的影响

影响主要体现在生成前校验：

1. 如果基础门槛未满足，则阻止进入依赖 foundation 的生成流程。
2. 如果基础门槛已满足，但中级建议不足，则给提醒。
3. 用户确认继续后，按现有方式继续生成。

### 10.3 对 AI 协作链路的影响

故事基底内部的 AI 协作不应受门槛限制。

也就是说：

- 用户可以在基础层未完成时就继续使用 foundation AI 协作。
- 门槛主要作用于“依赖 foundation 的外部生成入口”。

### 10.4 对持久化与迁移的影响

由于 `StoryFoundation` 结构不变，因此：

1. 现有快照格式可以继续使用。
2. 不需要新增迁移脚本。
3. 老数据在普通模式下也能正常显示。
4. 如果老数据里高级层已有值，普通模式只需给出状态提示即可。

## 11. 实施顺序建议

### 阶段 A：设计元数据落地

1. 定义模块层级映射。
2. 定义基础门槛字段。
3. 定义建议字段。
4. 输出统一的 readiness 计算函数。

### 阶段 B：界面最小改造

1. `FoundationPanel` 增加普通模式 / 高级模式切换。
2. 模块列表按层级可见性过滤。
3. 扩展编辑区只在高级模式下显示。
4. 模块卡片增加层级标记与字段标签。

### 阶段 C：生成前提醒接入

1. 在依赖 foundation 的生成入口前接入 readiness 检查。
2. 未达门槛时阻止进入。
3. 达到门槛但建议不足时弹提醒对话框。

### 阶段 D：进度与帮助文案调整

1. 更新 `FoundationProgressBar` 的文案。
2. 更新帮助文档与模块说明。
3. 补充普通模式下对高级层已填内容的提示。

---

## 12. 验收标准

满足以下条件，可视为 v3.1 设计落地完成：

1. 普通模式下，用户默认只看到基础层和中级层。
2. 高级模式下，用户可以查看并编辑全部层级。
3. `StoryFoundation` 数据结构保持不变。
4. 基础层达到最低门槛后，用户可以直接进入生成流程。
5. 中级建议不足时，系统会先弹提醒对话框，而不是直接阻止。
6. 高级层为空时不阻塞生成，高级层有值时仍然参与生成。
7. 普通模式下若已有高级内容，界面能给出清楚提示。
8. 现有 foundation 持久化、AI 协作、prompt 注入链路不因分层而失效。

---

## 13. 最终结论

v3.1 的核心不是减少能力，而是**重新安排能力的出现顺序**。

它希望达成的状态是：

1. 新用户先用最少信息开始。
2. 需要更多控制时，再进入中级与高级。
3. 系统用轻门槛和软提醒替代重门槛和强拦截。
4. 底层模型保持稳定，界面与校验规则逐步增强。

对当前项目而言，这条路线比直接删字段、缩 schema 或重做 foundation 数据模型更稳，也更适合后续继续演进。
